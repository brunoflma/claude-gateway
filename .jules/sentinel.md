## 2026-06-21 - [HIGH] Fix Overly Permissive CORS Configuration
**Vulnerability:** The local proxy reflected the user's `Origin` header to any value, combined with setting `access-control-allow-credentials: 'true'`. This meant that an attacker controlling `malicious.com` could trick a user's browser into sending requests to the local proxy (`https://localhost:8443`), and read the responses, potentially bypassing the browser's Same-Origin Policy and extracting sensitive information or utilizing the proxy's credentials/access.
**Learning:** Local proxies often use wildcard or reflected CORS to allow connection from applications like Office Add-ins running on local or specific domains. However, reflecting the `Origin` without validation is a critical security vulnerability, as it bypasses the browser's `Private Network Access` protections or general cross-origin protections when combined with credentials.
**Prevention:** Always use an explicitly defined whitelist of allowed origins (e.g., `*.office.com`, `localhost`) rather than blindly reflecting the `Origin` header. Fallback to a safe default like `'null'` if the origin does not match the whitelist.
